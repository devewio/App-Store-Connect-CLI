# publish

> High-level publishing workflows for TestFlight, plus App Store release guidance

The `publish` command remains the high-level surface for TestFlight workflows. For App Store releases, use `asc release run` to manage version setup, metadata application, build attachment, validation, and submission in one deterministic pipeline.

## Usage

```bash  theme={null}
asc publish testflight [flags]
```

## Workflows

### TestFlight workflow

Upload and distribute to TestFlight:

```bash  theme={null}
asc publish testflight \
  --app APP_ID \
  --ipa /path/to/MyApp.ipa \
  --group "External Testers"
```

This workflow:

1. Uploads the build
2. Waits for processing
3. Distributes to specified beta groups

### Canonical App Store workflow

```bash  theme={null}
# Upload a build first
asc builds upload --app APP_ID --ipa /path/to/MyApp.ipa

# Then run the App Store release pipeline
asc release run \
  --app APP_ID \
  --version 1.2.0 \
  --build BUILD_ID \
  --metadata-dir ./metadata/version/1.2.0 \
  --confirm
```

This workflow:

1. Ensures or creates the target version
2. Applies metadata and localizations
3. Attaches the selected build
4. Validates readiness
5. Submits for review

## TestFlight flags

<ParamField path="--app" type="string" required>
  App ID for the build
</ParamField>

<ParamField path="--file" type="string" required>
  Path to IPA file
</ParamField>

<ParamField path="--groups" type="string">
  Comma-separated beta group names or IDs

  Example: `"External Testers,Internal Team"`
</ParamField>

<ParamField path="--wait" type="boolean">
  Wait for build processing to complete

  Default: `true`
</ParamField>

## App Store release flags

<ParamField path="--app" type="string" required>
  App ID for the version
</ParamField>

<ParamField path="--version" type="string" required>
  Version string (e.g., `1.2.0`)
</ParamField>

<ParamField path="--build" type="string" required>
  Build ID to attach before validating and submitting
</ParamField>

<ParamField path="--metadata-dir" type="string" required>
  Metadata directory to apply during the release pipeline
</ParamField>

<ParamField path="--confirm" type="boolean" required>
  Required confirmation flag for a mutating App Store release
</ParamField>

<ParamField path="--platform" type="string">
  Platform: `IOS`, `MAC_OS`, `TV_OS`, or `VISION_OS`
</ParamField>

## Example workflows

### Publish to TestFlight

```bash  theme={null}
asc publish testflight \
  --app 123456789 \
  --ipa build/MyApp.ipa \
  --group "External Testers,Beta Team"
```

### Stage an App Store release without submitting

```bash  theme={null}
asc release stage \
  --app 123456789 \
  --version 1.2.0 \
  --build BUILD_ID \
  --copy-metadata-from 1.1.9 \
  --dry-run
```

### Submit to the App Store

```bash  theme={null}
asc release run \
  --app 123456789 \
  --version 1.2.0 \
  --build BUILD_ID \
  --metadata-dir ./metadata/version/1.2.0 \
  --confirm
```

### CI/CD integration

```yaml  theme={null}
# GitHub Actions
- name: Publish to TestFlight
  run: |
    asc publish testflight \
      --app ${{ secrets.APP_ID }} \
      --ipa MyApp.ipa \
      --group "External Testers"
```

## Error handling

If any step fails, the workflow stops and reports the error:

```
Release failed: metadata directory is missing required localization files
```

You can retry individual steps using the lower-level commands:

```bash  theme={null}
asc builds upload --app 123456789 --ipa MyApp.ipa
asc submit status --version-id VERSION_ID
```
