# submit

> Submission lifecycle operations; App Store publishing uses `release run`

The `submit` group now focuses on submission lifecycle operations after a release has been staged or submitted. For an end-to-end App Store publish, use `asc release run`. For a pre-submit staging pass that stops before review, use `asc release stage`.

## Usage

```bash  theme={null}
asc submit <subcommand> [flags]
```

## Canonical App Store publish

```bash  theme={null}
asc release run --app APP_ID --version VERSION_STRING --build BUILD_ID --metadata-dir ./metadata/version/VERSION_STRING --confirm
asc release run --app 123456789 --version 1.2.0 --build BUILD_ID --metadata-dir ./metadata/version/1.2.0 --confirm
```

1. Validates the version is ready
2. Applies metadata and localizations
3. Attaches the selected build
4. Creates and submits the review submission

## Check submission status

```bash  theme={null}
asc submit status --id SUBMISSION_ID
asc submit status --version-id VERSION_ID
```

## Cancel an active submission

```bash  theme={null}
asc submit cancel --id SUBMISSION_ID --confirm
asc submit cancel --version-id VERSION_ID --app APP_ID --confirm
```

## Example workflows

### Stage before you submit

```bash  theme={null}
asc release stage --app 123456789 --version 1.2.0 --build BUILD_ID --copy-metadata-from 1.1.9 --dry-run
asc validate --app 123456789 --version 1.2.0
```

### CI/CD release

```yaml  theme={null}
# GitHub Actions
- name: Run App Store release pipeline
  run: |
    asc release run --app ${{ secrets.APP_ID }} --version ${{ github.ref_name }} --build "${{ env.BUILD_ID }}" --metadata-dir "./metadata/version/${{ github.ref_name }}" --confirm
```

### Monitor the review lifecycle

```bash  theme={null}
asc submit status --version-id VERSION_ID
asc submit cancel --version-id VERSION_ID --app 123456789 --confirm
```
